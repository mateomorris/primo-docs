    How to use SvelteKit with Primo

    SvelteKit is a full-stack framework that makes it possible to build any kind of web application (Primo itself is built with SvelteKit!). But when it comes to managing site content, you're usually stuck with storing your page content in Markdown files and trying to train nontechnical collaborators how to use Git (something they're never a fan of).

    By combining SvelteKit with Primo, you can unlock the full potential of both tools to get the power and full-stack flexibility of SvelteKit alongside the buttery smooth content editing, page building, and collaboration of Primo.

    When to use SvelteKit with Primo

    If you're currently using SvelteKit, you can easily add Primo to your project to enable content editing and page building for a subset of your pages or routes, or just manage your content from Primo and everything else from SvelteKit. If you're using Primo, you can use it alongside SvelteKit to enable app-like functionality in your codebase. For example:

    • Manage your marketing pages in the same codebase as your application code and enable easy content collaboration for them.

    • Set up a couple SvelteKit cloud functions to collect form submissions, manage e-commerce functionality, or relay an AI API response.

    When to use SvelteKit or Primo

    In general, we recommend only using Primo when you have a site without many pages (i.e. fewer than 50) that uses simple presentational Blocks and doesn't require any backend functionality. And we recommend *only *using SvelteKit when you have a web application (i.e a functional website) that doesn't require a significant amount of content management, only contains content in a single language, and doesn't require content management by non-technical users.

    Methods

    Note that you can combine aspects from one method with another, depending on your project requirements, and you can always adapt them as needed. Here we outline four primary methods for combining SvelteKit with Primo - which one you choose will depend on the requirements of your given project.

    Primo as a Headless CMS

    Use when you only want to manage your content from Primo and everything else from your SvelteKit codebase.

    Fetching site data

    You can use Primo to manage your page content and render them from SvelteKit by using the Primo REST API (currently in Alpha) or directly querying your content from your site's Supabase database.

    REST API
    fetch('https://yourprimoserver.com/api/thesiteurl') // home page
  .then(res => res.json())
  .then(json => {
    console.log(json)
  })

fetch('https://yourprimoserver.com/api/thesiteurl/thepageurl') // child page
  .then(res => res.json())
  .then(json => {
    console.log(json)
  })
    Query from Database
    // initialize supabase above (https://supabase.com/docs/reference/javascript/initializing)
let { data: pages, error } = await supabase
  .from('pages')
  .select(*, site!inner(url))
  .match({
    'url: 'thepageurl',
    'site.url': 'thesiteurl',
  })

    Previewing with SvelteKit components

    Typically, Headless CMSs are decoupled from the presentation of your website since they're only concerned with your content (meaning that you have to edit your content without knowing what it will look like on your site). But with Primo, you can import your SvelteKit components (see 'Primo using a SvelteKit component library' below) in order to manage that content in the full context it will appear when it gets built out to your SvelteKit site.

    Primo frontend + SvelteKit backend

    Use when you want to manage everything from Primo, but need to add some backend functionality. You can use Primo as-is and give it a backend for all kinds of stuff using SvelteKit's cloud function functionality.

    Primo frontend + SvelteKit backend starter kit.

    SvelteKit codebase serving Primo pages

    Use when you have an application and want some subset of routes to serve Primo pages managed from a Primo server.

    For example - you might create a Primo site for your SvelteKit application's marketing pages so you can easily manage them from your Primo Server (separate from your application code) but still serve them from the same domain (i.e. not have to serve your marketing pages from the main domain and your application from the 'app.' subdomain.

    SvelteKit + Primo pages example repo.

    Primo using a SvelteKit component library

    Use when you have some components that would better be managed externally than from Primo's built-in code editor. Typically, this will be useful for maintaining a personal component library of reusable elements that you can share across all your Primo sites with things like buttons, forms, etc. You can also use this method to manage components that outgrow Primo like highly interactive applications, data tables, etc. (including version control).
